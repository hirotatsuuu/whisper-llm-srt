# AIモジュール仕様・運用マニュアル

本プロジェクト（whisper-llm-srt）で採用している3つのAI・自然言語処理ライブラリの連携メカニズム、仕様、および管理方法について解説します。

---

## 1. 全体アーキテクチャ（一本道パイプライン）

本システムは、データの安全性と正確性を担保するため、以下の「3階層モジュール設計」を完全な一本道で処理します。途中でデータの順序やタイムスタンプのバインディング（紐付け）が壊れないよう設計されています。

1. **【耳・下準備】OpenAI Whisper (transcriber.py)**
   - 音声データを解析し、ミリ秒単位のタイムスタンプ付きセグメント（塊）に切り分けます。
   - 直後に、発話時間を狂わせないためにフィラー（「ええと」「あの」等）を空文字（`""`）へと置換する下処理を行います。
2. **【頭脳】Ollama / ELYZA (llm_checker.py)**
   - 動画全体の文字起こしから「全体のあらすじ（文脈）」を最初に構築します。
   - あらすじと専門用語辞書をコンテキストに含め、最大10行ずつのバッチ（グループ）単位でローカルLLMへ送信し、元の時間インデックス（ID）を維持したまま、文脈に沿った誤字脱字・固有名詞の補正を行います。
3. **【デザイナー】Google BudouX (formatter.py)**
   - 綺麗になったテキストを、機械学習モデルを用いた文節解析によって、日本語として不自然ではない位置で細かく再分割します。
   - 最終的に、1行あたり「10文字以上〜20文字以内」のテロップ最適なサイズにミリ秒単位で時間を伸縮させ、SRTファイルとして出力します。

## 2. 各AIモジュールの詳細仕様

### 2.1. OpenAI Whisper (音声認識)
- **役割**: 音声からテキストへの変換、および正確なタイムスタンプ（開始時間・終了時間）の切り出し。
- **モデルサイズの特徴**:
  - `tiny` / `base` (標準): 動作が非常に軽く、CPU環境でも高速に動作します。一般的な会話の文字起こしには十分な性能を持ちます。
  - `small` / `medium`: 認識精度が向上しますが、相応のメモリ（VRAM/RAM）を要求します。
  - `large`: 業界最高峰の精度ですが、処理が重いためグラフィックボード（GPU）環境での実行を推奨します。
- **補足**: 本システムでは、Whisperの実行時に `dictionary.txt` から読み込んだ単語を `initial_prompt` として渡すことで、第1段階での固有名詞の聞き間違いを強力に防止しています。

### 2.2. ELYZA / Llama 3 (文脈校正・LLM)
- **役割**: 前後の文脈を解釈した誤字脱字の修正、および専門用語辞書ベースの表記統一。
- **採用モデル**: `hf.co/mmnga/Llama-3-ELYZA-JP-8B-gguf`
- **特徴**: Metaの「Llama 3 (8B)」をベースに、日本のAIスタートアップ「ELYZA」が追加学習を施した超高性能な日本語特化モデルです。
- **制御マニュアル**:
  - LLMの動作やモデル名は `src/llm_checker.py` の最上部にある `LLM_MODEL_NAME` で一元管理されています。
  - 万が一、LLMの応答が通信エラーやメモリ不足で途絶えた場合、システムは自動的に **「安全対策（生存ルート）」** へ切り替わり、該当ブロックの文字データをWhisperの生データのまま保護してクラッシュを防ぎます。

### 2.3. Google BudouX (レイアウト整形)
- **役割**: 独自の軽量な機械学習モデル（文字の並びの予測）を使い、日本語を「単語・文節の途中」でぶつ切りにすることなく、美しい改行・分割ポイントを見つけ出します。
- **特徴**: 従来の形態素解析器（MeCabなど）のように巨大な辞書を必要としないため、プログラムが軽量で、Windows/Macなどの環境を問わず常に安定して高速動作します。

## 3. 日常の運用・チューニング手順

### 固有名詞のアップデート方法
動画内で頻出する人名、商品名、最新のIT用語（例: Vite, Pinia, ChatGPTなど）がある場合は、実行前に **`data/dictionary.txt`** に追記してください。
- **書き方**: 1行に1単語ずつ、正確な表記を記述します。
- **効果**: ここに登録された単語は、Whisperの聞き取りプロンプトと、LLM（ELYZA）への「絶対に守るべき正解リスト」の両方に完全同期して共有されるため、校正精度が劇的に向上します。

## 4. AI関連のトラブルシューティング

### Q1. LLMの処理でフリーズする・パソコンが極端に重くなる
- **原因**: 8B（約80億パラメータ）のモデルをCPUだけで動かしている場合、一時的にメモリ（RAM）やCPU使用率が100%に達し、処理が遅くなることがあります。
- **対策**:
  1. 実行中は、他にメモリを大量に消費するブラウザ（Chromeのタブ等）や動画編集ソフトを一度閉じてください。
  2. どうしてもスペックが不足する場合は、`src/llm_checker.py` の最上部にあるモデル名を、Google製の軽量モデル（例: `"gemma2:2b"` など ※あらかじめ `ollama pull gemma2:2b` が必要）に書き換えることで、劇的に動作を軽量化できます。

### Q2. LLMが勝手に字幕の文字を要約・削減してしまう
- **原因**: LLMが「テロップを綺麗にする」という命令を「短く分かりやすくまとめる」と誤解して解釈してしまった場合に発生します。
- **対策**: `src/llm_checker.py` 内のプロンプト（命令文）に、「意味を変えないこと」「文字を勝手に省略せず、誤字脱字の修正のみに留めること」という制約をより強固に追記してチューニングしてください。

### Q3. 「Error: pull model manifest」というエラーが消えない
- **原因**: Ollamaでモデルを指定する際の名前が間違っているか、通信が遮断されています。
- **対策**: 本システムで指定しているELYZAのフルパス（`hf.co/mmnga/Llama-3-ELYZA-JP-8B-gguf`）が、ターミナルおよび `src/llm_checker.py` の定数と完全に一致しているか、スペースなどが混ざっていないかを確認してください。
